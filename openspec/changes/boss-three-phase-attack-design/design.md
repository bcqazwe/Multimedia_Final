# Design: boss-three-phase-attack-design

## 目標
將現有 Boss 擴充為一個三階段戰鬥對象。Boss 總血量為 3000，並以每 1000 HP 為一個階段：
- Phase 1: 3000 ~ 2001
- Phase 2: 2000 ~ 1001
- Phase 3: 1000 ~ 1
- HP 歸零後進入勝利結算

設計重點是讓後段階段更具侵略性，而不是只提升傷害數值。

## 現況與約束
現有原型已具備以下條件：
- `main.py` 管理遊戲狀態、血條渲染與結算畫面。
- `Boss.py` 已負責 Boss 影像、位置與水平隨機移動。
- `Health.py` 已提供血量顯示與分段式 Boss 血條。

本次設計應該延續這個結構，不破壞目前的 HUD、勝利畫面與返回主選單流程。

## Boss 狀態機
Boss 建議使用以下核心狀態：
- `ENTER`: 進場或初始定位
- `IDLE`: 短暫等待或微調位置
- `TELEGRAPH`: 顯示攻擊前搖
- `ATTACK`: 實際發動攻擊
- `RECOVER`: 攻擊後短暫冷卻
- `PHASE_TRANSITION`: 轉階段演出
- `DEFEATED`: 血量歸零

階段切換以 HP 門檻判定，攻擊池則依階段權重選擇。

## 三階段戰鬥節奏
### Phase 1: 教學與讀招
目的：讓玩家看懂 Boss 的節奏。
- 前搖較長。
- 攻擊間隔較大。
- 彈幕密度低，安全區明顯。
- Boss 大多維持在畫面上半部中間附近。

### Phase 2: 壓力與混合
目的：增加走位壓力與同時處理多威脅的能力。
- 前搖縮短。
- 攻擊間隔縮短。
- 同時存在的投射物數量增加。
- Boss 會更常朝玩家所在半邊偏移。

### Phase 3: 侵略與收尾
目的：讓玩家感受到 Boss 進入狂暴或失控狀態。
- 前搖最短。
- 攻擊間隔最短。
- 攻擊之間銜接更緊密，幾乎沒有長空窗。
- Boss 優先封鎖玩家退路，而不是只做正面壓制。

## 九種攻擊設計
建議將 9 招拆成 3 個攻擊家族，每階段各選 3 招，但參數與節奏不同。

### 攻擊家族 A: 彈幕型
1. Phase 1 - 直線掃射
   - Boss 朝下方中央發射慢速直線子彈。
   - 彈道規律，方便玩家認知。
2. Phase 2 - 交叉扇形
   - 一次發射多顆，左右角度分散。
   - 讓玩家開始必須移動到安全扇區。
3. Phase 3 - 高速鎖定彈
   - 開火前先鎖定玩家當前位置。
   - 子彈速度更快，且可連發兩輪。

### 攻擊家族 B: 位移型
4. Phase 1 - 慢速橫移
   - Boss 偶爾往左或往右滑動一次，動作單純。
5. Phase 2 - 短衝接回位
   - Boss 快速衝向半邊，再回到偏中間位置。
   - 常與彈幕或落點攻擊連用。
6. Phase 3 - 連段突進
   - Boss 連續兩次短衝，第二次更快。
   - 目的是逼迫玩家改變站位。

### 攻擊家族 C: 場地壓制型
7. Phase 1 - 單點落彈
   - 在玩家附近落下一個警示區，延遲爆炸。
8. Phase 2 - 雙點交錯落彈
   - 兩個落點交錯出現，限制走位路線。
9. Phase 3 - 區域封鎖大招
   - 以更高頻率生成多個危險區，短時間封鎖大範圍安全區。
   - 這招不必一定高傷，但要很有壓迫感。

## 侵略性遞增原則
後段階段的「更有侵略性」應反映在以下參數：
- 前搖時間更短。
- 攻擊間隔更短。
- 彈速更快。
- 同時存在的危險物更密集。
- Boss 位置更偏向玩家半邊，減少純中線站位的安全性。
- Recovery 時間更短，讓 Boss 不會顯得停頓。

## 與現有程式的整合方式
### `Boss.py`
- 維持 Boss 座標、大小與渲染責任。
- 新增 phase、attack_state、attack_timer、cooldown_timer 等欄位。
- 新增攻擊決策與執行介面，例如 `update(now_ms, player_state)`。

### `main.py`
- 根據 Boss HP 決定目前階段。
- 在每幀呼叫 Boss 更新與攻擊生成。
- 延續現有血條與 WIN 畫面流程。
- 當 Boss 歸零時，停止 Boss 攻擊與渲染，顯示結算畫面。

### `Health.py`
- 維持 3000 HP 的 Boss 血條分段顯示。
- 以 1000 HP 為單位顯示 `X3 / X2 / X1`。

## 平衡建議
- Phase 1 以「看得懂」為主，寧可慢一點。
- Phase 2 以「開始混亂」為主，但保留可預測規律。
- Phase 3 以「壓迫感」為主，但仍要保留少量固定安全窗口，避免完全不可玩。

## 風險與對策
- 風險：9 招過多，實作會分散。
  - 對策：先實作 3 個攻擊家族，再透過參數與階段變體擴成 9 招。
- 風險：後段太難導致玩家感受差。
  - 對策：用更短前搖與更密集節奏取代單純高傷。
- 風險：Boss 與血條、結算、碰撞耦合過深。
  - 對策：以狀態機與明確介面拆分責任，避免 `main.py` 失控膨脹。
